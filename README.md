# Document Intelligence Platform

A Retrieval-Augmented Generation (RAG) system that lets you upload documents and ask questions about them in natural language. Answers are generated by Claude and grounded strictly in the source document — with the exact passages used shown alongside every answer, and every interaction recorded to an audit log.

Built as an end-to-end, production-minded portfolio project: not a notebook demo, but a structured application with a web interface, separation of concerns, traceability, and a clear roadmap toward production concerns like observability and scaling.

---

## What it does

Upload a PDF, ask a question, get an answer that comes **only** from that document. If the answer isn't in the document, the system says so rather than inventing one.

- **Ask questions in plain English** — "What are the MLOps challenges?" returns a structured answer drawn from the uploaded document.
- **Grounded, not hallucinated** — Claude is constrained to answer only from retrieved context, with an explicit "I could not find that in the document" fallback.
- **Source transparency** — every answer displays the exact document passages it was built from, so any claim can be verified.
- **Single-document focus** — uploading a new document cleanly replaces the previous one, so answers never mix across sources; a clear button resets the store from the UI without touching the terminal.
- **Audit trail** — every question, answer, source set, and timestamp is logged for traceability and review.

## How it works

The system follows the standard RAG pattern, split into two phases: **ingestion** (done once per document) and **query** (done per question).

```
Ingestion:   PDF ──> parse ──> chunk ──> embed ──> store (vector DB)

Query:       Question ──> embed ──> search vector DB ──> top chunks
                                                            │
                                          chunks + question ▼
                                                   Claude (generation)
                                                            │
                                                   Grounded answer + sources
```

**Ingestion pipeline**
1. **Parse** — extract raw text from the PDF (PyMuPDF).
2. **Chunk** — split the text into overlapping pieces so no idea is lost at a boundary.
3. **Embed** — convert each chunk into a vector that captures its meaning (local sentence-transformers model).
4. **Store** — save the vectors in a persistent vector database (ChromaDB).

**Query pipeline**
1. **Embed the question** using the same model as the chunks, so they're comparable.
2. **Search** the vector database for the chunks closest in meaning to the question.
3. **Generate** — pass those chunks plus the question to Claude, constrained to answer only from the provided context.
4. **Return** the answer along with the source chunks, and log the interaction.

The key design property: the **same embedding model** encodes both documents and questions, which is what allows a question to match relevant passages even when the wording is different.

## Tech stack

| Layer | Choice | Why |
|-------|--------|-----|
| LLM (generation) | Anthropic Claude (Haiku) | Strong grounded answers; Haiku keeps cost low for development |
| Embeddings | sentence-transformers (`all-MiniLM-L6-v2`) | Runs locally — free, offline, no extra API key |
| Vector store | ChromaDB (persistent) | Lightweight, local, purpose-built for semantic search |
| Document parsing | PyMuPDF | Reliable PDF text extraction |
| Web UI | Streamlit | Fast to build, keeps the whole app in Python |
| Backend structure | FastAPI-ready service layer | Separation of concerns; UI is a thin layer over reusable services |

## Architecture

The project separates **presentation** from **logic** — the UI calls reusable backend services, it doesn't contain business logic itself.

```
doc-intelligence-platform/
├── backend/
│   ├── services/
│   │   ├── ingestion.py     # parse PDFs into text
│   │   ├── retrieval.py     # store chunks + semantic search (ChromaDB)
│   │   ├── llm.py           # Claude generation with anti-hallucination prompt
│   │   └── audit.py         # log every interaction
│   └── utils/
│       ├── chunker.py       # split text into overlapping chunks
│       ├── embedder.py      # local embedding model
│       └── config.py        # centralized config from environment
├── frontend/
│   └── app.py               # Streamlit web UI (upload + Q&A)
├── .streamlit/
│   └── config.toml          # UI theme
├── requirements.txt
└── README.md
```

This structure means each piece is independently testable, and adding a new capability (e.g. a new document format) is a localized change rather than a rewrite.

## Key design decisions

**Local embeddings over a cloud embedding API.** Embeddings run on-device using sentence-transformers — no API key, no per-call cost, works offline. The tradeoff is a one-time model download and slightly less "managed" infrastructure, but for a self-contained system this is the cleaner choice. The architecture keeps embedding behind a single module, so swapping to a cloud provider later is a one-file change.

**Grounding to prevent hallucination.** The generation prompt instructs Claude to answer *only* from the retrieved context and to explicitly say when the answer isn't present. This is what separates a trustworthy document-QA system from a chatbot that fills gaps with plausible-sounding invention.

**Source transparency.** Returning the exact chunks used — not just the answer — makes every response verifiable. This matters for any real-world use where answers need to be trusted or audited.

**Audit logging for traceability.** Every interaction is recorded with a timestamp, the question, the answer, and the sources used. In a real deployment, this is what lets you answer "what did the system say, when, and based on what?" — essential for debugging, compliance, and accountability.

**Retrieval tuning, validated against ground truth.** Early testing on a table-heavy document surfaced incomplete answers. Tracing the issue showed it was a **retrieval** problem (the relevant content hadn't been retrieved), not a **generation** problem (the model answered honestly from what it was given). Increasing the number of retrieved chunks recovered the missing content. This "is it retrieval or generation?" diagnostic is central to debugging RAG quality.

**Clean document switching.** Testing with multiple documents revealed that uploading a second file mixed its content with the first, because chunks accumulated in the vector store with colliding IDs and answers blended across sources. The fix explicitly clears stored records before ingesting a new document, so each query runs against exactly one source. This surfaced a broader lesson: vector store state management is as important as the retrieval logic itself, and bugs there are easy to mistake for model failures.

## Getting started

### Prerequisites
- Python 3.12+
- An Anthropic API key ([platform.claude.com](https://platform.claude.com))

### Setup

```bash
# Clone the repository
git clone https://github.com/barun-khan/doc-intelligence-platform.git
cd doc-intelligence-platform

# Create and activate a virtual environment
python3.12 -m venv venv
source venv/bin/activate        # On Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt
```

### Configuration

Create a `.env` file in the project root:

```
ANTHROPIC_API_KEY=your_key_here
LANGSMITH_API_KEY=your_langsmith_key_here   # optional, for observability
LANGSMITH_PROJECT=doc-intelligence-platform
APP_ENV=development
```

### Run

```bash
streamlit run frontend/app.py
```

Then open the local URL shown in the terminal (typically `http://localhost:8501`). Upload a PDF in the sidebar, then ask questions in the main panel.

## Roadmap

Planned enhancements, in rough priority order:

- **LLM observability (LangSmith)** — trace every Claude call's latency, token usage, and cost. Integration is designed and wired into config; full activation is a fast-follow.
- **Audit log viewer** — a dedicated page to browse interaction history in the UI, rather than reading the log file directly.
- **Leaner audit records** — store chunk references (IDs) instead of full chunk text to keep the log compact at scale.
- **Multi-format ingestion** — extend beyond PDF to `.docx`, `.html`, and others. The architecture separates parsing from the pipeline, so each format is just a new parser feeding the same chunk→embed→store flow.
- **Structure-aware chunking** — split on natural boundaries (paragraphs, sections, tables) instead of fixed character counts, to keep structured content like tables intact.
- **Polished frontend** — an optional React interface as a presentation layer once the backend is feature-complete.

## Notes

This project was built to demonstrate end-to-end RAG system design — document ingestion, semantic retrieval, grounded generation, and the operational concerns (traceability, document management, evaluation) that distinguish a real application from a prototype. It is a portfolio project rather than a production deployment; the roadmap above outlines what closing that gap would involve.
